# 91-7_component_refactoring.md

■ 1. 目的
Services.tsx の肥大化を解消し、ロジック（表示条件）とデザイン（HTML/CSS）を分離する。
将来のデザイン委託時に、デザイナーが「表示データ（Props）」を理解し、安全に装飾を変更できる構造を構築する。

---

■ 2. 新規作成コンポーネントおよび設計（Props）

● src/components/ServiceCard.tsx
- 役割: サービスプランの1カード分の表示。
- 引数（Props）: 
  - id: プラン識別子（予約ボタンのリンク生成に使用）
  - name: プラン名
  - description: 説明文
  - price: 数値（toLocaleStringで整形前）
  - taxText: 「税込」「税別」の表記
  - durationText: 「60分」「1回」等の期間表記
  - isAvailable: 販売中かどうかのフラグ（ボタンの活性/非活性制御用）

● src/components/SlotItem.tsx
- 役割: 予約可能日時の1行分の表示。
- 引数（Props）:
  - date: 日付文字列
  - time: 時間範囲文字列
  - status: 内部状態（'open' | 'full' | 'closed'）
  - statusText: 表示用ラベル（'受付中' 等）

---

■ 3. 実装の進め方（安全第一プロトコル）

1. [Componentの新規作成]
   src/components/ 配下にファイルを作成。冒頭にデザイナー向けの「Component説明コメント」を付与する。
   ※最初は既存のインラインスタイルをそのまま移植し、表示の同一性を担保する。

2. [二重表示テスト（重要）]
   Services.tsx において、古いインラインHTMLを消さずに、その直後に作成したコンポーネントを配置する。
   ブラウザで新旧の表示を並べ、余白（Margin/Padding）やフォントサイズに差異がないか目視で厳密に比較する。

3. [旧コードのクリーンアップ]
   表示の完全一致を確認後、古いインラインHTMLコードを削除し、コンポーネント呼び出しのみに集約する。

---

■ 4. デザイナーへの「優しさ」の仕込み（コード冒頭の定型）

各コンポーネントファイルの先頭に、以下の形式で注釈を記述する。

// @component: [コンポーネント名]
// @description: [役割の簡潔な説明]。デザイナーはこのファイル内のHTML/CSSを自由に編集可能です。
// @props: [受け取るデータ一覧]（ロジック層から自動的に注入されます）

---

■ 5. 期待される効果
- ページ全体の構造（骨組み）が1画面に収まる分量になり、メンテナンス性が向上する。
- 状態（statusやisAvailable）に応じたデザイン変更を、コンポーネント内で完結して行えるようになる。
