# システム計画書：高信頼予約決済基盤(v1.8)

---

## 1. プロジェクトの目的と設計思想

Cloudflare Workers + D1 + Stripe を活用し、予約枠の「仮確保」から「確定」までを最短時間で回す、将来の多拠点展開を見据えた高信頼・高効率な基盤を構築する。

| 設計前提 | 内容 |
|---|---|
| DB構成 | シングル D1 インスタンス運用 |
| 時刻管理 | 内部処理は **UTC**、表示・検索は **JST** で完全分離 |
| 状態管理 | セッション不使用。**URL-Driven Architecture** を採用 |

---

## 2. 業務フロー：高回転・自己修復型トランザクション

### 2.1 プラン別フロー定義（正典）

ユーザーがどのボタンを押した時に、どの「状態」へ遷移するかの全体像。

| プラン | カレンダー | 遷移フロー | 決済 |
|---|---|---|---|
| 単発・スポット（決済型） | **必須** | カレンダー選択 → URLパラメータ付与 → Stripe決済 | Stripe Checkout |
| 顧問契約（リード獲得型） | **不要** | 直接フォーム（Resend経由） → 手動調整 | なし（オフライン調整） |

### 2.2 ステートレス設計の宣言（URL-Driven Architecture）

セッションを使わず、URLパラメータで情報を引き回す設計を採用する。

~~~
/services?plan=spot&slot=slot_001&date=2026-04-07
~~~

**採用理由**

| 理由 | 内容 |
|---|---|
| 複数タブ対策 | セッション共有による予約の混線を物理的に排除 |
| Honoとの相性 | ステートレスなエッジ環境と設計思想が一致 |
| デバッグ容易性 | URLを見るだけで状態が把握可能 |

---

## 3. インベントリ・ロック（Stripe仕様への完全同調）

同時押し（レースコンディション）を物理的に排除し、Stripeのセッション期限と同期させる。

~~~sql
UPDATE slots
SET status = 'pending', expires_at = ?
WHERE id = ? AND tenant_id = ? AND status = 'available'
-- 更新件数 0 なら失敗と判定
~~~

| 期限 | 設定値 | 理由 |
|---|---|---|
| Stripe セッション期限 | **31分（1860秒）** | API引数 `expires_at` で指定 |
| DB 仮確保期限 | **35分（2100秒）** | Stripe側が先に失効することを保証し、二重予約をゼロにする |

---

## 4. Webhook優先・Cron補完のハイブリッド自浄作用

APIレート制限（N+1問題）を回避しつつ、データの整合性を維持する。

| 経路 | トリガー | 処理内容 |
|---|---|---|
| メイン（Webhook） | `checkout.session.completed` | 予約を確定 |
| メイン（Webhook） | `checkout.session.expired` | 枠を即時開放 |
| 安全網（Cron / 5分間隔） | 期限切れ `pending` 抽出 | 冪等性テーブル確認後、未処理の「迷子」のみ Stripe API へ照合・強制失効 |

---

## 5. データベース・スキーマ設計

SQLite の特性を活かし、将来の「大工事」を避けるための「空き箱（プレースホルダー）」を設置する。

### ① 在庫管理テーブル (`slots`)

| カラム名 | 型 | 役割・規約 |
|---|---|---|
| `tenant_id` | TEXT (INDEX) | 必須。初期値: `taiga_shizen`。将来のマルチテナント用 |
| `id` | TEXT (PK) | 枠の一意識別子 |
| `start_at_unix` | INTEGER | 開始時刻（Unix Timestamp / UTC） |
| `date_string` | TEXT (INDEX) | JST固定日付（`YYYY-MM-DD`）。カレンダー検索用 |
| `status` | TEXT | `available` / `pending` / `booked` |
| `expires_at` | INTEGER | 仮確保の期限（Unix Timestamp / UTC） |
| `slot_duration` | INTEGER | 枠の粒度（分）。将来の可変時間対応用 |

### ② 決済冪等性テーブル (`processed_events`)

`event_id`（PK）をキーとし、二重処理を完全に遮断する。

---

## 6. 実装規約（エンジニア・監査向けデバッグ指針）

| リスク項目 | 対策 | 期待される効果 |
|---|---|---|
| JST/UTC逆転 | `date_string` 保存時は必ず JST（`Asia/Tokyo`）に変換 | 深夜・早朝の予約が「前日」にズレるバグを封鎖 |
| 二重予約防止 | 31分(Stripe) < 35分(D1) の階層構造を維持 | 「払ったのに枠がない」事故を物理的に排除 |
| API負荷制限 | Reconciliation 前に冪等性テーブルを走査 | 数万件の運用でも Stripe API 制限に抵触しない |
| 検索速度低下 | `date_string` と `tenant_id` への複合 INDEX | カレンダー表示のフルスキャンを回避 |
| 将来の相乗り | 全クエリに `tenant_id` を含める設計 | Phase 3 への移行コストを 90% 削減 |
| URLパラメータ汚染 | サーバー側でパラメータを必ずバリデーション | 不正な `slot_id` や `plan` 値による誤処理を防止 |

## 8. なんでもメモ(後日マージの可能性)
・日本時間の「午前0時〜午前9時」の間に予約操作した場合、前日の日付になる可能性  

・単発プランが「Book Now（予約する）」、顧問契約は「Inquire / Contact（相談する）」等、分けるか、または統一的な名称にするかを検討する。 

・URLパラメータの「長さ制限とエンコード」は、ブラウザやサーバーのURL長制限（約2000文字）に抵触する可能性があるため、自由コメントは1000文字以内など制限をかける。
