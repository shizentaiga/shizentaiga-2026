# 91-5_git_branch_workflow.md：ブランチ運用ワークフロー

---

## 1. VS Code でのブランチ視認

エディタの画面構成を利用して、常に自分がどのブランチを操作しているか意識する習慣をつける。

| 項目 | 内容 |
|---|---|
| 確認場所 | VS Code 画面左下の「ステータスバー」 |
| `[main]` 表示 | 本番世界（公開用）を操作中 |
| `[test-branch]` 表示 | 開発・テスト世界（隔離環境）を操作中 |
| 操作 | ステータスバーをクリック → ブランチ切り替えメニューが開く |

---

## 2. ブランチ運用：標準ワークフロー

パラレルワールドを作成し、変更を記録して、クリーンに削除するまでの一連の流れ。

### Step 1：新しいブランチの作成と移動

未コミットの変更を保持したまま、新しいブランチを作成して移動する。

~~~sh
git checkout -b test-branch
~~~

### Step 2：変更の記録（コミット）

テスト世界の中でファイルの修正を確定させる。

~~~sh
git add README.md
git commit -m "docs: テスト内容の追記"
~~~

### Step 3：リモート（GitHub）への送信

GitHub 上にもブランチを作成し、内容をアップロードする。

~~~sh
git push origin test-branch
~~~

### Step 4：本番世界への帰還

作業が終わったら、必ず本番ブランチに戻る。

~~~sh
git checkout main
~~~

### Step 5：不要になったブランチの削除

~~~sh
# ローカルから削除（未マージの場合は -D で強制削除）
git branch -d test-branch

# リモート（GitHub）から削除
git push origin --delete test-branch
~~~

---

## 3. ブランチ運用の黄金律

| # | 原則 | 内容 |
|---|---|---|
| 1 | `main` は神域 | 本番ブランチで直接作業しない。必ずブランチを切ってから開始する |
| 2 | 小刻みなマージ | テストが通ったら速やかに本番へ統合し、古いブランチは即削除。コンフリクトを防ぐ |
| 3 | プレビューの活用 | Cloudflare 等はブランチごとに確認用 URL が自動発行される。本番反映前に最終確認する |
