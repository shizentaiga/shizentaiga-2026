# 00_Core_Strategic_Philosophy_and_Grid_Model

## 1. Executive Summary: The "North Star" Objective
本プロジェクトの究極の目的は、既存の予約SaaSでは到達し得ない「極限の信頼性」を備えた予約インフラを構築することである。
SNSやライブコマースによる瞬間的なアクセス集中（秒間数万アクセス）においても、物理的に重複を許さない世界基準の堅牢性を実現し、事業者の機会損失と信頼失墜を構造的にゼロにする。

## 2. Core Architecture: The Grid Inventory Model (チップ・グリッド・モデル)
時間を「線」ではなく、30分単位（デフォルト）の「チップ（原子）」として管理する。

### 2.1 物理的な排他制御
- **原子化**: 在庫を「スタッフID × チップ開始時間」という一意のレコードとして定義する。
- **DB制約による防御**: 予約（Slots）が発生した際、対象のチップを1対1で占有する。DBレベルの一意制約（Unique Key）により、計算を介さず物理的にダブルブッキングを封じ込める。
- **アトミック確保**: 複数チップにまたがる予約（例: 90分プラン）は「全か無か」の原則で一括確保し、不完全な予約状態を排除する。

### 2.2 運用のハイブリッド戦略
- **供給（スタッフ側）**: スケジュールはチップ単位で登録。重複を許容する柔軟な登録インターフェースにより、スタッフ自身が「隙間時間」をコントロール可能な設計とする。
- **需要（ユーザー側）**: 予約時に「プラン所要時間 ＋ 事後バッファ（片付け時間）」を合算したチップ数を消費する。

## 3. Business Protocol: Strategy for Scalability & Reliability
### 3.1 収益性と市場展開
- **収益モデル**: 予約ごとのトランザクション課金または月額課金。顧客の成長がシステムの収益に直結する。
- **市場ターゲット**: 予約開始と同時に枠が埋まる人気美容師、インフルエンサー、高単価サービス提供者。
- **横展開**: 同一の設計思想で、チケット販売、イベント予約、医療予約などの巨大市場へ対応可能。

### 3.2 リスク管理の原則
- **Pending & TTL**: 決済中の仮予約には厳密な有効期限（10〜15分）を設定し、在庫の「幽霊ロック」を防ぐ。
- **5%ルール**: 年間売上の5%以上に相当する障害リスクが発生した場合、基盤改修を最優先する経営判断原則。
- **輻輳規制**: アプリケーションおよびUI層で秒間アクセスを監視し、DB負荷を閾値以下に抑える「水際対策」を徹底する。

## 4. Implementation Guidelines by Layer
- **UI Layer**: 在庫状況の高速表示と、エッジ側でのアクセスフィルタリング。
- **Workers Layer**: チップ算出、決済プロセスの同期、トランザクション発行。
- **DB Layer (D1)**: 一意制約による真実の保持。インデックス検索のみによる超高速な空き枠判定。
