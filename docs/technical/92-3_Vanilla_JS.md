# 技術資料：コンポーネント間連携における Vanilla JS の役割と実装

## 1. Vanilla JS の概要
Vanilla JS（バニラ JS）とは、特定のフレームワーク（React, Vue, jQueryなど）を使用せず、ブラウザ標準の機能のみを使って記述された純粋な JavaScript を指します。

### 本システムでの採用理由
- **軽量・高速**: ライブラリの読み込みが不要なため、Cloudflare Workers 等のエッジ環境で極めて高速に動作します。
- **HTMX との相性**: HTMX が HTML 要素を動的に書き換える際、標準的なイベントリスナーを用いる Vanilla JS は、複雑なライフサイクル管理なしに要素の状態を監視できます。

---

## 2. Vanilla JS の実装方法（一般論）
モダンな Vanilla JS の実装では、以下の 3 つのステップでコンポーネント間の疎結合な連携を実現します。

1.  **イベントの発火 (Dispatch)**: 
    ある要素の状態が変わった際、`CustomEvent` を作成して全体に通知します。
2.  **イベントの購読 (Subscribe)**: 
    別の要素（あるいはグローバルな監視役）が `addEventListener` でその通知を待ち受けます。
3.  **伝播の制御 (Bubbling)**: 
    `bubbles: true` を設定することで、HTML 構造の親階層へ通知を拡散させ、どこからでも検知可能にします。

---

## 3. 今回の具体的な実装事例
本予約システムでは、「カレンダーでの日付選択」と「予約ボタンの有効化」を同期させるためにこの仕組みを使用しています。

### A. 通知の送信側（CalendarSection.tsx）
カレンダーの日付（セル）がクリックされた際、HTMX の処理と並行して「日付が選ばれた」という事実をシステム全体に知らせます。

**実装イメージ:**
(※HTML内のインラインスクリプトとして記述)
document.dispatchEvent(new Event('selectionChange', { bubbles: true }));

### B. 通知の受信側（BookingFooter.tsx）
画面下部に固定されているフッターは、カレンダーがどこにあるかを知る必要はありません。ただ「通知」が届くのを待機します。

**実装イメージ:**
document.addEventListener('DOMContentLoaded', () => {
  // 1. 通知を待ち受ける
  document.addEventListener('selectionChange', () => {
    // 2. 通知が来たら「プラン・日付・時間」の選択状況を確認しに行く
    syncSelection(); 
  });
});

### C. この設計によるメリット
- **依存性の排除**: カレンダー側は「誰がこの通知を受け取るか」を知る必要がありません。フッターを削除したり、別のコンポーネントに変更したりしても、カレンダー側のコードを修正する必要がない「疎結合」な状態が保たれます。
- **デバッグの容易性**: ブラウザのコンソールから `monitorEvents(document)` 等を実行することで、通知が正しく飛んでいるかを簡単にテストできます。

---

## 4. プログラマーへの注意点
- **二重登録の防止**: `DOMContentLoaded` 外でリスナーを登録すると、HTMX の遷移（ページの部分書き換え）の仕組みによってはリスナーが重複して登録される可能性があります。常に `document` や `window` に対して一度だけ登録する「イベント委譲（Event Delegation）」を推奨します。
- **イベント名の命名規則**: 本システムでは、選択状態の変更に対して `selectionChange` という名称を統一して使用してください。
