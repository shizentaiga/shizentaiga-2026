# フロントエンド設計要件
## デザイン・プログラム分離設計

---

## 1. 設計思想：三層分離と「型」による契約

将来の外部デザイン委託およびマルチデバイス展開を見据え、以下の三層を論理的に分離することを理想とする。  
ただし、開発初期段階では**スピードを最優先**し、コードの肥大化に合わせて段階的に切り出しを行う。

~~~
[ Visual Layer ]   src/components/   →  デザイン委託の主戦場
[ Logic Layer  ]   src/pages/        →  初期実装の拠点
[ Domain Layer ]   src/domain/       →  UIに依存しない業務ロジック
~~~

---

## 2. システムアーキテクチャ（フォルダ構成）

| フォルダ / ファイル | 役割・責務 | ポイント |
|---|---|---|
| `src/types/` | データ・UIの型定義 | `domain.ts`（コア）と `view.ts`（UI用）に分離 |
| `src/domain/` | 純粋な業務ロジック | 料金計算・バリデーション等。UIに依存しない |
| `src/constants/` | 共有定数・スタイル | `business.ts`（固定値）、`style.ts`（デザイン共通値） |
| `src/components/` | Visual Layer | Tailwind CSS を用いた UI 部品。デザイン委託の主戦場 |
| `src/pages/` | Logic & Entry Layer | **【初期実装の拠点】** 最初はここに全てを記載して良い |
| `src/debug/` | サンドボックス | **【ルール】** 外部API・DBアクセス禁止。静的確認用のみ |

---

## 3. 実装・リファクタリングの運用指針

### 3.1 段階的な切り出し（理想と現実の調整）

- 初期実装時は `pages/Services.tsx` 等にロジックとデザインを同居させ、最短での疎通を目指す。
- コードが数十〜100行程度の段階で無理に細分化する**オーバーエンジニアリングは避ける**。
- 「固定値」「ロジック」「デザイン」という大きな単位での分離が必要になったタイミングで、順次リファクタリングを検討する。

### 3.2 Tailwind CSS：動的クラス名の原則禁止

文字列結合はビルド時にスタイルが消えるリスクがあるため禁止。**フルネーム指定を徹底する。**

~~~typescript
// NG：ビルド時にスタイルが消える
className={`text-${color}-500`}

// OK：フルネーム指定
className="text-red-500"
~~~

### 3.3 型とマッピングの責務

データの加工が必要になった際は `hooks/` 等へ切り出し、Pages（表示側）が DB の複雑な構造を直接意識しない設計を維持する。

---

## 4. セキュリティと秘匿性の担保

| 項目 | 対策 |
|---|---|
| Preview 環境の D1 遮断 | `wrangler.toml` にて本番とプレビューのバインディングを分け、本番 DB へのアクセスを物理的に不能にする |
| サンドボックス運用 | デザイナーにはモックデータのみを公開し、機密情報の漏洩リスクをゼロにする |

---

## 5. 将来の拡張性

`domain/` や `types/` が整理されていれば、将来のスマホアプリ化の際にも**ロジックをそのまま流用**し、**UIだけを差し替える**ことが容易になる。
